## Setup & Commands

This project uses `uv` for dependency management (Python 3.11).

```bash
# Install dependencies
uv sync

# Run the MCP server directly
uv run python main.py

# Run via the installed script entry point
uv run bigquery-mcp
```

GCP authentication must be set up before running:
```bash
gcloud auth application-default login
```

Set `GCP_PROJECT_ID` in `.env` (see `.env` for reference). SMTP credentials (`SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD`) are also required for email delivery.

## Architecture

The entire server lives in `main.py` as a single-file [FastMCP](https://github.com/jlowin/fastmcp) server running over `stdio` transport.

**BigQuery client** (`get_bq_client`) is a thread-safe singleton built with Application Default Credentials. It is lazy-initialised on first tool call.

**Use cases** are YAML files under `use_cases/`. Each file describes a named query template with parameters. They are loaded fresh on every tool call (no in-process caching) via `_load_use_cases()`.

**MCP tools exposed:**

| Tool | Purpose |
|---|---|
| `list_tables` | Returns all table IDs registered in `accessible_tables.yaml` |
| `get_table_schema` | Reads locally cached schema from `table_schemas/<table_id>.yaml` |
| `sync_table_schemas` | Scans use cases for new table IDs → updates `accessible_tables.yaml` → fetches live schema for all tables → writes `table_schemas/` |
| `list_use_cases` | Returns all use case summaries (id, name, description, keywords, parameters) |
| `search_use_cases` | Token-match search against use case names/descriptions/keywords |
| `get_use_case` | Returns full YAML detail including `query_template` |
| `execute_query` | Runs arbitrary SQL and returns JSON rows (≤1000 rows) |
| `generate_csv_from_query` | Runs SQL, writes CSV to `output/` with a timestamp filename, returns path + stats |
| `send_csv_via_email` | Sends a CSV file as an attachment via SMTP |

**Intended LLM workflow:** search use cases for business context/patterns + list tables and fetch schemas for column details → compose a custom query → `generate_csv_from_query` → `send_csv_via_email`.

**Three sources of knowledge:**
- `accessible_tables.yaml` — registry of allowed tables (fully-qualified IDs only)
- `table_schemas/<table_id>.yaml` — locally cached schema per table; populated by `sync_table_schemas`, gitignored (generated)
- `use_cases/` — business-context examples showing metric definitions, grouping patterns, and query idioms for specific questions

## accessible_tables.yaml

A flat list of fully-qualified BigQuery table IDs the server is allowed to query. No schema is stored here.

```yaml
tables:
  - project.dataset.table_name
```

`get_table_schema` rejects any table not in this list.

## table_schemas/

One YAML file per table, named `<table_id>.yaml`. Generated by `sync_table_schemas` — do not edit manually. Each file contains `table_id`, `num_rows`, `synced_at`, and a `columns` list with `name`, `type`, `mode`, and `description` fields. Run `sync_table_schemas` whenever a table's schema changes.

## Use Case YAML Schema

```yaml
id: unique_snake_case_id
name: Human Readable Name
description: >
  What this use case analyzes.
keywords:
  - keyword1
  - keyword2
tables:
  - project.dataset.table_name
parameters:
  - name: param_name
    description: What this parameter means
query_template: |
  SELECT ...
  WHERE date BETWEEN '{param_name}' AND '{other_param}'
```

The `tables` field lists every fully-qualified BigQuery table referenced in `query_template`. When `sync_table_schemas` runs, any table ID listed here that is absent from `accessible_tables.yaml` is added automatically before schemas are fetched.

Parameters are substituted using Python `str.format()` style `{param_name}` placeholders inside `query_template`.
